# Architecture & Development Principles

Этот документ описывает как мы строим проект — структуру, принципы и договорённости.
Читать всем участникам перед тем как писать код.

---

## Общая структура

```
backend/      Python + FastAPI
frontend/     TypeScript + React + Vite
docs/         документация и контракты
tests/        тесты
```

Frontend и backend — независимые приложения. Общаются только через HTTP API.

---

## Backend (Python + FastAPI)

### Структура

```
backend/
├── main.py          # точка входа: создание app, подключение роутеров, middleware
├── <module>/
│   ├── __init__.py
│   ├── router.py    # FastAPI-роутер модуля (если есть HTTP-эндпоинты)
│   ├── service.py   # бизнес-логика
│   └── models.py    # Pydantic-модели (запрос/ответ)
```

### Принципы

**Слои не смешиваются.** `main.py` только подключает роутеры. Роутер только принимает запрос и вызывает сервис. Вся логика — в сервисе.

**Типизация через Pydantic.** Все входящие и исходящие данные описываются моделями. Никаких голых `dict` на входе или выходе эндпоинта.

**Зависимости через DI.** Использовать `Depends()` для инъекции зависимостей — не создавать объекты внутри функций напрямую.

**Модули независимы.** Модуль не импортирует другой модуль напрямую. Взаимодействие — только через интерфейсы (функции с явными сигнатурами), не через внутренности.

**Конфиг через окружение.** Все настройки — из `.env` через `python-dotenv`. Никаких захардкоженных значений в коде.

**Ошибки обрабатываются явно.** Каждый сервис при ошибке либо возвращает безопасное значение (например пустой список), либо поднимает `HTTPException` с понятным сообщением. Голые `except: pass` запрещены.

## Frontend (TypeScript + React + Vite)

### Структура

Фронтенд организован по **фичам (feature-based)**. Каждая фича — отдельная папка, которая содержит всё что к ней относится: компоненты, хуки, запросы, типы. Общий код вынесен в `shared/`.

```
frontend/src/
├── features/                  # модули по функциональности
│   └── <feature-name>/
│       ├── components/        # компоненты этой фичи
│       ├── hooks/             # хуки этой фичи
│       ├── api.ts             # запросы к backend
│       └── types.ts           # типы запросов/ответов
├── shared/                    # всё переиспользуемое между фичами
│   ├── components/            # общие UI-компоненты (кнопки, инпуты и т.п.)
│   ├── hooks/                 # общие хуки
│   └── types/                 # общие типы
├── App.tsx                    # роутинг
└── main.tsx                   # точка входа
```

Пример: если есть фича `catalog` и фича `chat` — они живут независимо. `catalog` не импортирует из `chat`.

### Принципы

**Фича — самодостаточна.** Всё что нужно фиче лежит внутри её папки. Не надо искать по всему проекту.

**Из `features/` импортируем только наружу, не внутрь друг друга.** Если двум фичам нужно одно и то же — выносим в `shared/`.

**Компонент делает одно.** Если компонент превышает ~100 строк или решает более одной задачи — разбить. Имя компонента должно точно описывать что он рендерит.

**Типы описываются явно.** Никаких `any`. Все пропсы типизированы. Ответы от API описаны в `types.ts` фичи.

**Запросы только через `api.ts` фичи.** Никаких `fetch` прямо в компонентах. Все обращения к backend — через функции в `api.ts`, которые возвращают типизированные данные.

**Состояние минимально.** `useState` — для локального состояния. Если состояние нужно нескольким компонентам внутри фичи — хук. Между фичами — только через `shared/` или поднятие в `App.tsx`.

## API-контракт между frontend и backend

- Все эндпоинты начинаются с `/api/`
- Запросы — `application/json`
- Ответы — `application/json`
- Ошибки возвращают `{ "detail": "сообщение" }` и соответствующий HTTP-статус
- Контракты (типы запросов и ответов) описываются в `docs/` и дублируются в `frontend/src/types/`

## Общие договорённости

**Язык комментариев** — русский.

**Ветки** — см. [tasks/README.md](tasks/README.md).

**Коммит** — одна логическая единица изменений. Если можно разбить — разбиваем. Сообщение коммита: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`.

**PR** — обязательно описание: что сделано и зачем. Минимум 1 ревью перед мержем.

**Хардкод запрещён** — конфиги, ключи, URL — только через переменные окружения или конфиг-файлы.

**.env не коммитится** — только `.env.example` с заглушками.
